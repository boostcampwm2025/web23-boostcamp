PR 작성 요청이 들어오면 아래 규칙을 반드시 따른다.
	1.	템플릿 준수
	•	리포지토리의 .github 디렉터리에 있는 PR 템플릿/양식을 먼저 확인한다.
	•	PR 본문은 해당 템플릿의 구조, 섹션, 문구 스타일을 최대한 그대로 따른다.
	2.	작성 위치 고정
	•	PR 본문 초안은 항상 리포지토리 루트의 pr_description.md 파일에 작성한다.
	•	다른 파일이나 채팅 메시지로 PR 본문을 대신 작성하지 않는다.
	3.	맥락 반영
	•	대화/작업 로그/변경 사항을 기반으로 이전 맥락과 의사결정 배경을 반영한다.
	•	“왜 이 변경이 필요한지”, “무엇이 달라지는지”, “대안 검토가 있었는지”까지 포함해 서술형으로 충분히 설명한다.
	4.	작업 내용 구체화
	•	실제 변경된 작업을 빠짐없이 반영한다.
	•	아래 항목을 가능한 한 세세하게 채운다:
	•	변경 범위(모듈/기능/경로)
	•	핵심 구현 내용
	•	API/스키마/설정 변경 여부
	•	부수 효과 및 영향 범위(호환성, 성능, 보안, 배포)
	•	테스트 방법/결과(재현 절차 포함)
	•	스크린샷/로그/근거(가능한 경우)
	•	롤백/마이그레이션/주의사항
	5.	빈칸 최소화 원칙
	•	“N/A”로 처리하기 전에, 실제로 적을 수 있는 정보가 있는지 먼저 확인한다.
	•	정보가 부족하면 “추정”이 아니라, 현재 확보된 사실 기준으로 최대한 상세히 작성하고, 불확실한 부분은 명확히 표시한다(예: TBD, 확인 필요).
